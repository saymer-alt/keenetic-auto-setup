# 04 — watchdog

Почему система не требует постоянного ручного перезапуска.

---

## Задача

Не мониторинг ради мониторинга, а:

> **автоматическое восстановление Mihomo при проблемах**

Без участия человека.

---

## Почему не "while true"

Плохой вариант:

```bash
while true; do
  check
  sleep 5
done
```

Проблемы:

* жрёт CPU
* не даёт системе «спать»
* на слабых роутерах → лаги

---

## Решение

```id="cronflow"
cron → каждые 5 минут → watchdog → завершился
```

* скрипт живёт ~10–20 секунд
* не нагружает систему
* работает предсказуемо

---

## Где находится

```bash
/opt/etc/cron.5mins/mihomo_watchdog
```

Запуск:

```bash
*/5 * * * * root /bin/sh /opt/etc/cron.5mins/mihomo_watchdog
```

---

## Основные параметры

```bash id="params"
PROXY_URL=127.0.0.1:7890
CHECK_URLS="http://google.com/generate_204 http://connectivitycheck.gstatic.com/generate_204"
WAN_CHECK_URL=http://1.1.1.1
COOLDOWN=300
LOCK_FILE=/tmp/mihomo_watchdog.lock
```

---

## Что именно проверяется

### 1. Жив ли процесс

```bash id="check1"
pidof mihomo
```

Если нет:

```id="action1"
→ restart
```

---

### 2. Cooldown (защита от циклов)

```bash id="check2"
NOW - LAST < 300
```

Если недавно был рестарт:

```id="action2"
→ ничего не делать
```

---

### 3. Есть ли интернет (WAN)

```bash id="check3"
curl http://1.1.1.1
```

Если нет:

```id="action3"
→ skip
```

👉 важно:

> не рестартить Mihomo, если проблема у провайдера

---

### 4. Работает ли прокси

```bash id="check4"
curl -x socks5://127.0.0.1:7890 ...
```

Проверяется 2 URL:

* google.com/generate_204
* gstatic generate_204

---

## Почему именно generate_204

* возвращает HTTP 204 (без тела)
* минимальный трафик
* быстро
* почти не блокируется

---

## Логика результата

| Код | Значение                      |
| --- | ----------------------------- |
| 204 | всё ок                        |
| 000 | таймаут / нет ответа          |
| 502 | прокси жив, но upstream мёртв |
| 403 | Google заблокировал           |

---

## Когда происходит рестарт

```id="logic"
если:
  mihomo жив
  И cooldown прошёл
  И WAN есть
  НО proxy check fail

→ restart
```

---

## Jitter (разнос по времени)

```bash id="jitter"
sleep $(( $(date +%s) % 25 ))
```

---

## Зачем это нужно

Если у тебя:

* 10–20 роутеров
* один сервер

Без jitter:

```id="ddos"
все одновременно → запрос → пик нагрузки
```

С jitter:

```id="spread"
запросы распределены на 0–25 сек
```

---

## Почему не RANDOM

BusyBox:

* может не иметь RANDOM
* или всегда возвращает 0

👉 `date +%s` работает везде

---

## Логирование

Файл:

```bash
/opt/var/log/mihomo_watchdog.log
```

Примеры:

```id="log1"
proxy fail [000/000], restart
```

```id="log2"
WAN unreachable, skip
```

```id="log3"
OK
```

---

## Ротация логов

```bash id="rotate"
tail -n 100
```

Если лог > ~120 строк:
→ оставляет последние 100

---

## Почему не logrotate

* лишняя зависимость
* не везде есть
* сложнее поддержка

---

## Ручной запуск (очень важно)

```bash id="debug"
sh -x /opt/etc/cron.5mins/mihomo_watchdog
```

Показывает:

* jitter
* коды ответов
* cooldown
* решения

👉 основной инструмент дебага

---

## Частые проблемы

### `proxy fail [000/000]`

Причина:

* Mihomo не слушает порт
* или завис

---

### `proxy fail [502/502]`

Причина:

* сервер VPN недоступен

---

### `WAN unreachable`

Причина:

* интернет реально пропал

---

### лог пустой

Причина:

* cron не работает

Проверка:

```bash
ps | grep cron
```

---

## Важные ограничения

* watchdog чинит **только Mihomo**
* не чинит:

  * VPN сервер
  * DNS
  * провайдера

---

## Что можно менять

```bash id="tune"
COOLDOWN=300
CHECK_URLS=...
PROXY_URL=...
```

---

## Что НЕ трогать

* порядок проверок
* lock-файл
* jitter

---

## Главное

Watchdog — это:

> **не мониторинг, а механизм самовосстановления**
