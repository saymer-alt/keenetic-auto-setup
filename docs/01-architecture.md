# 01 — архитектура

Как это работает на самом деле.

Не «компоненты», а **потоки трафика и принятие решений**.

---

## Общая схема

```
Клиент (LAN)
      ↓
Keenetic
      ↓
Proxy0 (127.0.0.1:7890)
      ↓
Mihomo (Clash Meta)
      ↓
VPN (VLESS / Reality / etc.)
      ↓
Интернет
```

Параллельно:

```
VoIP (UDP 1400,3478,3482)
      ↓
bypass_wa (mark)
      ↓
VPN напрямую (мимо Mihomo)
```

И отдельно:

```
Watchdog
  ↓
проверка → рестарт Mihomo (если надо)
```

---

## Основная идея

Не «всё через прокси», а:

> **разные типы трафика идут разными путями**

---

## Потоки трафика

### 1. Обычный трафик (HTTP/HTTPS)

```
LAN → Proxy0 → Mihomo → VPN → Интернет
```

Почему так:

* Mihomo умеет правила (domains, geoip)
* можно гибко управлять маршрутизацией
* SOCKS5 универсален

---

### 2. VoIP (Telegram / WhatsApp)

```
LAN → iptables (mark) → bypass_wa → VPN → Интернет
```

Почему **в обход Mihomo**:

* VoIP = UDP + WebRTC
* через SOCKS5:

  * задержки
  * обрывы
  * иногда вообще не соединяется

👉 Поэтому:

> голос идёт напрямую через VPN, минуя прокси

---

### 3. Прямой трафик (опционально)

```
LAN → Keenetic → провайдер
```

Используется для:

* локальных ресурсов
* GEOIP private
* fallback

---

## Компоненты

### Mihomo (Clash Meta)

Роль:

* основной прокси
* rule-based routing
* работа с VLESS / Reality

Слушает:

```
127.0.0.1:7890 (SOCKS5)
```

---

### Proxy0 (Keenetic)

Связывает систему Keenetic с Mihomo:

* весь трафик с политики → в SOCKS5
* точка входа для маршрутизации

---

### MagiTrickle

Решает:

> какой домен куда отправить

Пример:

* `youtube.com` → через VPN
* `local` → напрямую

Работает через DNS и policy routing.

---

### bypass_wa (iptables)

Решает:

> как не убить голосовую связь

* маркирует UDP-трафик
* отправляет его в отдельную policy
* минует Mihomo

---

### S00ubifs

Решает:

> как не убить флешку

* переносит `/opt/var` в RAM
* убирает постоянные записи на диск

---

### Watchdog

Решает:

> что делать, когда всё сломалось

Проверяет:

1. жив ли процесс Mihomo
2. есть ли интернет (WAN)
3. отвечает ли прокси

Если нет:

```
→ рестарт Mihomo
```

---

## Почему не "просто VPN на роутере"

Стандартный сценарий:

```
LAN → VPN → Интернет
```

Проблемы:

* нет гибкой маршрутизации
* всё идёт через VPN
* VoIP страдает
* при падении VPN всё ломается

---

## Почему не "всё через Mihomo"

Пробовали. Не работает стабильно.

Проблемы:

* UDP (VoIP)
* задержки
* нестабильные звонки

---

## Почему именно такая схема

Потому что она:

* переживает падения прокси
* не ломает голос
* масштабируется (20+ роутеров)
* не убивает флешку

---

## Что происходит при сбое

### Сценарий 1: Mihomo умер

```
watchdog → pidof fail → restart
```

---

### Сценарий 2: VPN сервер недоступен

```
proxy check → fail → restart
```

(если проблема на сервере — не поможет, но фиксит подвисания)

---

### Сценарий 3: интернет пропал

```
WAN check → fail → ничего не делаем
```

👉 важно:

> не рестартить Mihomo, если проблема не в нём

---

## Сценарий "всё сломалось"

Что делать:

1.

```bash
/opt/etc/init.d/S99mihomo status
```

2.

```bash
cat /opt/var/log/mihomo_watchdog.log
```

3.

```bash
curl -x socks5://127.0.0.1:7890 https://ipinfo.io
```

---

## Главное

Это не «VPN на роутере».

Это:

> **система маршрутизации + самовосстановление + обход проблем сети**
